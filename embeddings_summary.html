01 What this file contains

Every row is one medical code. Every row has 107 columns: 7 metadata columns (Code, Type, Description, Category, Dollar_Weight, Volume_Weight, FWA_Composite) followed by 100 embedding dimensions (D001–D100).

The embedding vectors are L2-normalised to unit length (norm = 1.0000). This means you can compute the similarity between any two codes using a simple dot product — no division needed.

Important: ICD-10-CM codes are stored without dots (e.g. I2101 not I21.01), matching the CMS source file format. Strip dots before lookup if your claims data uses dotted notation.

02 The 10 semantic axes

The 100 dimensions are grouped into 10 meaningful axes of 10 dimensions each. Each axis encodes a distinct clinical or financial property. You can work with specific axes by slicing the vector rather than using all 100 dimensions — this is more powerful than full cosine similarity for targeted audits.

03 Knowledge sources encoded

These are structured knowledge bases whose clinical rules are statically encoded into each embedding — not raw data access. Every axis value is traceable to a specific source rule, making the embeddings auditor-explainable.

CMS IPPS FY2026

MS-DRG relative weights, geometric mean LOS, MCC/CC designation table (full Appendix C). Encoded into D011–D020 and D091–D100.

CMS NCCI FY2026

Column 1/2 edit pairs, add-on code flags, modifier indicators. 30 known bundling pairs encoded into D061–D070 with per-pair strength scores.

AHRQ CCS (Clinical Classification Software)

284 clinical categories mapped to domain axis D001–D010. Every ICD-10-CM code inherits its CCS group, overriding generic chapter defaults.

MIMIC-III/IV (published summaries)

Top-50 DX admission frequency, ICU rates, mean LOS statistics from published MIMIC papers. Encoded into D011–D020 (ICU rate → severity) and D091–D100 (LOS → financial weight).

AMA RVU (PFS FY2026)

Work RVU, total RVU, global surgery days for 25 high-volume CPT codes. Boosts D021–D030 (intensity) and D091–D100 (financial weight) for surgical procedures.

PubMed co-occurrence clusters

20 disease clusters from Elixhauser, Charlson, and AHRQ comorbidity studies (e.g. ACS cluster, sepsis cluster, heart failure cluster). Encoded as domain and severity boosts.

CMS OPPS FY2026

APC payment indicators, outpatient status indicators (T/S/X/N). Encoded into D051–D060 to distinguish inpatient vs outpatient billing profiles.

ICD-10 FY2026 source files

All 79,193 PCS codes from icd10pcs_order_2026.txt and 74,719 CM codes from icd10cm_codes_2026.txt. Real codes, real descriptions — not synthetically generated.

04 What you can do with it

①
Overpayment risk scoring on inpatient claims

Run every claim through the 12-step dimensional auditor (dim_auditor.py). Each step targets a specific axis range. Score 0–100; claims above 65 route to clinical audit queue. Inputs: claim ID, DX codes (split by PDX / MCC / CC / other SDX), procedure codes, LOS, DRG.

②
MCC/CC inflation detection

Compare the declared MCC/CC count to the embedding-supported count. Codes with D011 (severity) below 0.30 after L2 normalisation lack MCC-level support in the embedding space. MCC stacking = high intra-MCC cosine on D011–D020 (same-pathway coding).

③
NCCI bundling compliance

Component codes have D061 > 0.70; comprehensive codes have D061 < 0.20. If a claim contains two codes where one has D061 > 0.70 and the pair appears in the NCCI edit table, flag as bundling violation. Covers ECG components, colonoscopy subsets, lab panels, chemo add-ons, PT units, and more.

④
Medical necessity gap analysis

Compute cosine similarity between each procedure vector and each diagnosis vector on D071–D080 only (not full 100-dim cosine — that inflates scores). Pairs below 0.15 on the DX-Proc axis are medical necessity gaps. Flag procedures with no supporting diagnosis above threshold.

⑤
Anatomical site coherence

A cardiac procedure (D031 ≈ Chest) billed on a GI diagnosis claim (D031 ≈ Abdomen) will have near-zero cosine similarity on D031–D040. This catches cross-claim attribution errors, wrong-site surgery coding, and DRG manipulation via unrelated procedures.

⑥
DRG financial weight anomaly

High D091 procedures (CABG ≈ 0.85, THA ≈ 0.75) paired with low D091 diagnoses (UTI ≈ 0.15, wellness ≈ 0.05) signal DRG manipulation. The financial gap = mean(proc D091) − mean(DX D091). Gap > 0.40 = HIGH risk flag.

⑦
Upcoding and phantom billing signals

D081 = upcoding risk; D082 = unbundling risk; D083 = over-ordering; D084 = phantom billing. E&M codes (99215, 99291) have D081 > 0.18. CPAP and DME codes have D084 > 0.30. These are pre-computed signals — no threshold tuning needed to get started.

⑧
Weighted similarity search

Use medical_similarity.py to find the top-N most similar codes to any query code, with per-axis weight controls. Set fwa:3.0 to find codes with similar fraud patterns. Set clinical:0 severity:2.0 bundling:3.0 to find unbundling targets. Supports code comparison and top-N search.

05 How to load and query

Load once, keep in memory. At ~292 MB, the full matrix fits comfortably in RAM for 254,140 codes × 100 dimensions as float32 (~97 MB as a NumPy array). The CSV is the distribution format; convert to NumPy for production.

import csv, numpy as np

# Load into (N, 100) float32 matrix + lookup index
codes, vectors = [], []
with open("Medical_Embeddings_Master_2026.csv") as f:
    reader = csv.reader(f)
    hdr = next(reader)
    ds = hdr.index("D001")          # dimension start column
    for row in reader:
        codes.append(row[0])       # code string
        vectors.append(row[ds:ds+100])

matrix = np.array(vectors, dtype=np.float32)  # shape (254140, 100)
idx    = {c: i for i, c in enumerate(codes)}

# Similarity between two codes (full 100-dim)
def similarity(code_a, code_b):
    va = matrix[idx[code_a]]
    vb = matrix[idx[code_b]]
    return float(va @ vb)          # dot product = cosine (L2-normed)

# Axis-specific similarity (e.g. DX-Proc link, dims 70-79)
def axis_similarity(code_a, code_b, start, end):
    va = matrix[idx[code_a]][start:end]
    vb = matrix[idx[code_b]][start:end]
    na, nb = np.linalg.norm(va), np.linalg.norm(vb)
    if na == 0 or nb == 0: return 0.0
    return float((va / na) @ (vb / nb))

# Example: medical necessity link between STEMI and PCI
axis_similarity("I2101", "02703ZZ", 70, 80)   # → ~0.89 (high)
axis_similarity("N390",  "02703ZZ", 70, 80)   # → ~0.05 (low)

# Top-N similar codes to a query (on full vector)
def top_n(query_code, n=10):
    q = matrix[idx[query_code]]
    sims = matrix @ q                           # (N,) dot products
    sims[idx[query_code]] = -2               # exclude self
    top = np.argpartition(sims, -n)[-n:]
    return [(codes[i], sims[i]) for i in top[np.argsort(-sims[top])]]
Critical: Use axis-specific cosine (D071–D080) for medical necessity checks, not full 100-dim cosine. Full cosine between a CM code and a CPT code will be inflated (~0.85–0.99) because of shared structural dimensions. The DX-Proc link axis (70–79) isolates medical necessity signal.

medical_similarity.py
Interactive weighted similarity tool. Compare two codes, find top-N similar codes, adjust per-axis weights (e.g. boost FWA 3×, disable clinical domain). Interactive menu or CLI flags.
payment_integrity_auditor.py
7-step payment integrity audit with pairwise similarity matrices (DX×DX, DX×Proc, Proc×Proc). Includes MIMIC-aware severity validation and LOS vs DRG benchmark checks.
generate_test_cases.py
Generates 22 synthetic inpatient test cases across 12 categories (valid controls, MCC inflation, DX-Proc mismatch, unbundling, adversarial). Used to validate and tune audit thresholds.
overpayment_scorer.py
DRG-981 specific overpayment scorer. Flags claims where an extensive O.R. procedure is unrelated to the principal diagnosis. Seven sub-dimensions with configurable weights.

08 Best practices

  • TIP Use axis slices, not full cosine, for targeted audits. Full cosine between a DX and a proc code will be high (~0.85+) because shared structural dimensions dominate. Slice D071–D080 for medical necessity, D061–D070 for bundling, D031–D040 for anatomical coherence.
  • TIP Strip dots from ICD-10-CM codes before lookup. I21.01 → I2101. The source file (and therefore the embeddings) uses no-dot format matching CMS data exactly.
  • TIP Load the matrix into RAM as float32. 254,140 × 100 × 4 bytes ≈ 97 MB. For production, pre-compute the matrix once at startup. All similarity searches are then pure matrix–vector multiplications — fast enough for real-time claim adjudication.
  • TIP Use structured DX input (PDX / MCC / CC / other SDX) instead of a flat DX list. The dim_auditor.py and generate_test_cases.py support this schema. It enables Step 2 (severity validation) to compare declared MCC count against embedding-supported severity codes — the most clinically significant check.
  • TIP DRG/RVU values (D091) are relative, not absolute. High-DRG procedures (CABG, THA, MV) will have D091 ≈ 0.25–0.35 after normalisation, not 0.85. What matters is that CABG D091 > E&M D091 > Wellness D091. Use the ordering, not the absolute value, when writing thresholds.
  • TIP Use Dollar_Weight and FWA_Composite columns for quick pre-filtering. Before computing any cosine similarity, filter claims where any code has FWA_Composite > 0.25 — these are the highest-priority codes to examine. Dollar_Weight is a normalised proxy for CMS reimbursement tier.
  • TIP Tune thresholds on the 22 test cases before production. Run python generate_test_cases.py --run-audit and examine the alignment_check column in the results. Adjust the THRESHOLDS dict in dim_auditor.py until TC-001/002/003/021/022 (valid controls) score LOW and TC-006/010/013 (critical mismatches) score HIGH/CRITICAL.
  • NOTE Do not assume fraud — focus on documentation and coding inconsistencies. These embeddings detect patterns that are statistically inconsistent with the clinical narrative. Every flag should be reviewed by a clinical coder or auditor before action. The scores support clinical judgment — they do not replace it.
